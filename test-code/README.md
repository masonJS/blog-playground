## 테스트의 목표
> 지속 가능한 서비스, 프로젝트가 되게 하는게 목표이다.

과연 테스트 커버리지를 100%로 달성하는게 목표일까?

전혀 그렇지 않다고 생각한다.    
테스트 코드도 코드이다. 코드는 자산이 아니라 책임이다.    
코드를 작성함으로 내재적으로 잠재된 오류의 발생률이 높아질뿐만 아니라 코드의 유지관리비용이 증가한다.    
그리고 코드를 추가하거나 수정하는 비용도 늘어날 뿐이다.  
만약 테스트 코드의 목적을 수행하는데 가능한 적은 양의 코드로 작성할 수 있다면 그것이 최선의 해결책이다.   


## 테스트는 좋은 코드와 연관되어 있다.
좋은 코드는 "변경하기 쉬운"이라는 형용사를 가지고 있다.  
이 의미는 약한 결합도을 가지고 있는 코드를 뜻하며 반대로 강결합이 되어있는 코드는 유지비용이 증감되어 저품질코드로 분류되어진다.  
그렇다면 이러한 강결합으로 이루어진 코드를 테스트하기 쉬울까?  
당연히 매우 어렵다.  
그렇기때문에 좋은 코드는 항상 테스트 하기 쉬운 코드이다. 물론, 테스트하기 쉬운 코드가 모두 좋은 코드가 되는 것은 아니다.  
하지만 만약 내가 작성한 메인 코드가 테스트하기 어려운 코드라면 나쁜 코드 스멜이 난다는 지표로 봐도 무방하다고 생각한다.  


## given-when-then 패턴
테스트의 구조는 총 준비, 실행, 검증 3개의 구절로 나뉘어질 수 있다.  
이 3 구절중 제일 코드의 양이 많아질 가능성이 있는 구절은 준비 구간이다.  
준비 단게의 구절에서 너무 많은 양의 코드가 존재한다면 이부분을 좀더 코드 재사용의 목적으로 모듈화 해두면 좋다. (테스트 팩토리, 빌더, private 메소드)
