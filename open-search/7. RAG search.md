## RAG Search

RAG(Retrieval-Augmented Generation)는 LLM(대형 언어 모델)이 답변을 생성할 때, 미리 인덱싱된 문서에서 관련 정보를 검색(Retrieve)하여 컨텍스트로 주입한 뒤 답변을 생성(Generate)하는 패턴이다.

- RAG 의 기본 개념은 `1_ai_기초지식` 문서를 참고

### OpenSearch RAG 동작 흐름

```
사용자 질문
    │
    ▼
[OpenSearch] 관련 문서 검색 (kNN / lexical)
    │
    ▼
검색된 문서를 컨텍스트로 조합
    │
    ▼
[LLM (SageMaker / Bedrock 등)] 답변 생성
    │
    ▼
최종 응답 반환
```

OpenSearch는 `Search Pipeline` + `ML Commons` 플러그인 조합으로 이 흐름을 하나의 검색 요청 안에서 처리한다.

---

### Prerequisites

#### 1. 클러스터 설정 활성화

ML Commons의 메모리(대화 히스토리) 기능과 RAG 파이프라인 기능을 활성화한다.

```
PUT /_cluster/settings
{
  "persistent": {
    "plugins.ml_commons.memory_feature_enabled": true,
    "plugins.ml_commons.rag_pipeline_feature_enabled": true
  }
}
```

| 설정 키 | 설명 |
|---------|------|
| `memory_feature_enabled` | 대화 컨텍스트(메모리) 저장 기능 활성화. 멀티턴 대화에 필요 |
| `rag_pipeline_feature_enabled` | Search Pipeline 내 `retrieval_augmented_generation` 프로세서 사용 허용 |

> **버전 요구사항**: OpenSearch 2.12 이상, ML Commons 플러그인 2.12+ 필요

#### 2. 연결할 AI 모델 준비

RAG를 위해 답변 생성에 사용할 LLM을 OpenSearch Connector로 연결해야 한다. 아래 Development 섹션에서 Amazon SageMaker + DeepSeek-R1 모델을 기준으로 설명한다.

### Development

- Amazon SageMaker를 통해 호스팅한 DeepSeek-R1 모델을 LLM으로 사용
- 공식 가이드: [RAG using DeepSeek-R1 in Amazon SageMaker](https://docs.opensearch.org/latest/tutorials/gen-ai/rag/rag-deepseek-r1-sagemaker)

전체 구성 순서:
1. SageMaker 접근용 IAM Role 생성
2. OpenSearch Service에 IAM Role 구성
3. Connector 생성 (OpenSearch ↔ SageMaker 연결)
4. 모델 등록 및 배포
5. RAG Search Pipeline 구성
6. 실제 RAG 검색 쿼리 실행

#### [Step 1: Create an IAM role for Amazon SageMaker access](https://docs.opensearch.org/latest/tutorials/gen-ai/rag/rag-deepseek-r1-sagemaker/#step-1-create-an-iam-role-for-amazon-sagemaker-access)

OpenSearch가 SageMaker 엔드포인트를 호출할 수 있도록 IAM Role을 생성한다.

- Trust relationship: `es.amazonaws.com` (OpenSearch Service)가 해당 Role을 assume할 수 있어야 함
- 권한 정책: `sagemaker:InvokeEndpoint` 허용

---

#### [Step 2: Configure an IAM role in Amazon OpenSearch Service](https://docs.opensearch.org/latest/tutorials/gen-ai/rag/rag-deepseek-r1-sagemaker#step-2-configure-an-iam-role-in-amazon-opensearch-service)

Step 1에서 생성한 IAM Role ARN을 OpenSearch 도메인의 액세스 정책에 추가하여, OpenSearch 내부에서 해당 Role로 SageMaker를 호출할 수 있게 한다.

#### Step 3: Create a connector

OpenSearch와 SageMaker 사이의 연결(Connector)을 생성한다. Connector는 OpenSearch ML Commons가 외부 모델을 호출할 때 사용하는 설정 객체다.

**3-1. Step 2에서 생성한 IAM Role ARN으로 임시 AWS 자격증명 발급**

```bash
aws sts assume-role \
  --role-arn <step2 iam role arn> \
  --role-session-name <session custom name>
```

반환값 예시:
```json
{
  "Credentials": {
    "AccessKeyId": "ASIA...",
    "SecretAccessKey": "...",
    "SessionToken": "...",
    "Expiration": "2025-01-01T00:00:00Z"
  }
}
```

이 자격증명을 환경변수로 export하거나 `~/.aws/credentials`에 저장한다.

**3-2. 생성된 credential을 기반으로 Connector 생성 API 호출**

필요한 패키지 설치:
```bash
npm install @aws-sdk/credential-providers @smithy/protocol-http @aws-sdk/signature-v4 @aws-crypto/sha256-js axios
```

```ts
import { defaultProvider } from '@aws-sdk/credential-providers';
import { HttpRequest } from '@smithy/protocol-http';
import { SignatureV4 } from '@aws-sdk/signature-v4';
import { Sha256 } from '@aws-crypto/sha256-js';
import axios from 'axios';

// 1. 환경 변수 또는 ~/.aws/credentials에서 AWS 자격증명 자동 로드
const region = 'ap-northeast-2';
const host = ''; // opensearch domain endpoint (예: https://xxx.ap-northeast-2.es.amazonaws.com)
const service = 'es';

const payload = {
  name: 'DeepSeek R1 model connector',
  description: 'Connector for my Sagemaker DeepSeek model',
  version: '1.0',
  protocol: 'aws_sigv4',
  credential: {
    roleArn: '', // step 1 iam role arn
  },
  parameters: {
    service_name: 'sagemaker',
    region: region,
    do_sample: false,
    top_p: 0.01,
    temperature: 0.01,
    max_new_tokens: 512,
  },
  actions: [
    {
      action_type: 'PREDICT',
      method: 'POST',
      url: '', // 생성한 sagemaker endpoint url
      headers: {
        'content-type': 'application/json',
      },
      request_body:
        '{ "inputs": "${parameters.inputs}", "parameters": {"do_sample": ${parameters.do_sample}, "top_p": ${parameters.top_p}, "temperature": ${parameters.temperature}, "max_new_tokens": ${parameters.max_new_tokens}} }',
      // SageMaker 응답에서 generated_text를 추출하여 OpenSearch가 인식하는 형식으로 변환
      post_process_function: `if (params.result == null || params.result.length == 0) {
    throw new Exception('No response available');
  }

  def completion = params.result[0].generated_text;
  return '{' +
           '"name": "response",' +
           '"dataAsMap": {' +
              '"completion":"' + escape(completion) + '"}' +
         '}';`,
    },
  ],
};

// 2. AWS 자격증명 로드
const credentialsProvider = defaultProvider();
const credentials = await credentialsProvider();

// 3. HTTP 요청 객체 생성
const path = '/_plugins/_ml/connectors/_create';
const endpoint = new URL(host);
const request = new HttpRequest({
  protocol: endpoint.protocol,
  hostname: endpoint.hostname,
  port: endpoint.port ? Number(endpoint.port) : undefined,
  method: 'POST',
  path: path,
  headers: {
    'Content-Type': 'application/json',
    Host: endpoint.hostname,
  },
  body: JSON.stringify(payload),
});

// 4. SigV4 서명 (OpenSearch Service는 IAM 인증 필요)
const signer = new SignatureV4({
  credentials,
  service,
  region,
  sha256: Sha256,
});

const signedRequest = await signer.sign(request);

// 5. axios로 요청 전송
try {
  const response = await axios({
    method: 'POST',
    url: host + path,
    headers: signedRequest.headers,
    data: JSON.stringify(payload),
  });
  console.log(response.data);
} catch (err) {
  if (err.response) {
    console.error('Error:', err.response.status, err.response.data);
  } else {
    console.error('Error:', err.message);
  }
}
```

**3-3. 결과 확인**

`console.log(response.data)` 결과로 아래 형태의 응답이 반환된다:
```json
{ "connector_id": "<connector id>" }
```

이 `connector_id`는 Step 4에서 모델 등록 시 사용한다.

#### Step 5: Configure RAG

검색 결과를 LLM에 자동으로 전달하는 **Search Pipeline**을 생성한다. Search Pipeline은 검색 요청/응답을 전처리·후처리하는 OpenSearch의 파이프라인 기능이다.

**5-1. RAG Search Pipeline 생성**

```
PUT /_search/pipeline/rag_pipeline
{
  "response_processors": [
    {
      "retrieval_augmented_generation": {
        "tag": "rag_processor",
        "description": "RAG pipeline using DeepSeek R1",
        "model_id": "<step 4에서 배포한 model_id>",
        "context_field_list": ["text"],   // 검색된 문서에서 LLM 컨텍스트로 사용할 필드
        "system_prompt": "You are a helpful assistant. Answer the question based only on the provided context.",
        "user_instructions": "Generate a concise answer for the given question based on the search results."
      }
    }
  ]
}
```

| 옵션 | 설명 |
|------|------|
| `model_id` | Step 4에서 배포한 LLM 모델 ID |
| `context_field_list` | 검색된 문서 중 LLM에 전달할 필드명. 여러 개 지정 가능 |
| `system_prompt` | LLM에 전달할 시스템 지시 (선택) |
| `user_instructions` | 사용자 질문 앞에 붙는 추가 지시 (선택) |

**5-2. Pipeline 연결 확인**

```
GET /_search/pipeline/rag_pipeline
```

#### Step 6: RAG 검색 쿼리 실행

Search Pipeline을 사용하는 실제 RAG 검색 요청이다.

**6-1. 기본 RAG 검색**

```
GET /my-index/_search?search_pipeline=rag_pipeline
{
  "query": {
    "match": {
      "text": "OpenSearch의 장점은 무엇인가?"
    }
  },
  "size": 3,
  "_source": ["text", "title"],
  "ext": {
    "generative_qa_parameters": {
      "llm_question": "OpenSearch의 주요 장점을 설명해줘",
      "llm_model": "deepseek_r1",
      "context_size": 3,        // LLM에 전달할 검색 결과 수
      "message_size": 5,        // 대화 히스토리에서 참조할 메시지 수
      "timeout": 30             // LLM 응답 대기 시간(초)
    }
  }
}
```

응답 예시:
```json
{
  "hits": {
    "hits": [
      { "_source": { "title": "OpenSearch 개요", "text": "..." } },
      { "_source": { "title": "검색 기능", "text": "..." } }
    ]
  },
  "ext": {
    "retrieval_augmented_generation": {
      "answer": "OpenSearch의 주요 장점은 1) 오픈소스로 무료 사용 가능, 2) 강력한 전문 검색 기능, 3) 벡터 검색 지원으로 시맨틱 검색 가능..."
    }
  }
}
```

**6-2. kNN 벡터 검색 + RAG 조합 (Semantic RAG)**

텍스트 임베딩이 저장된 인덱스에서 의미 기반 검색 후 RAG를 적용한다:

```
GET /my-vector-index/_search?search_pipeline=rag_pipeline
{
  "query": {
    "knn": {
      "embedding": {
        "vector": [0.12, 0.34, ...],  // 질문을 임베딩한 벡터
        "k": 5
      }
    }
  },
  "_source": ["text", "title"],
  "ext": {
    "generative_qa_parameters": {
      "llm_question": "OpenSearch의 주요 장점을 설명해줘"
    }
  }
}
```

### 전체 구성 요약

```
[사용자 질문]
      │
      ▼
GET /index/_search?search_pipeline=rag_pipeline
      │
      ├─ 1. OpenSearch: 관련 문서 검색 (match / kNN)
      │
      ├─ 2. response_processor: retrieval_augmented_generation
      │       검색된 문서의 context_field_list 필드 → 컨텍스트 조합
      │
      ├─ 3. ML Commons → Connector → SageMaker 엔드포인트 호출
      │       (검색 컨텍스트 + llm_question 전달)
      │
      └─ 4. LLM 응답을 ext.retrieval_augmented_generation.answer 에 포함하여 반환
```

### 주의사항

- SageMaker 엔드포인트 호출 비용이 발생하므로, `size`와 `context_size`는 필요한 만큼만 설정한다
- `post_process_function`은 Painless 스크립트로 작성되며, LLM의 응답 구조가 변경되면 함께 수정해야 한다
- `memory_feature_enabled`를 활성화하면 대화 히스토리가 OpenSearch 내부 인덱스(`.plugins-ml-memory-*`)에 저장된다
- IAM Role의 신뢰 정책(Trust Policy)에 `es.amazonaws.com`이 포함되지 않으면 Connector 생성 시 권한 오류가 발생한다
