# AOS vs SOA — 데이터 레이아웃이 성능에 미치는 영향

## 1. 두 패턴이란?

대량의 데이터를 다룰 때, 같은 정보를 메모리에 배치하는 방식이 두 가지 있다.

### AOS (Array of Structures) — 객체 배열

데이터를 **"항목 단위"**로 묶어서 배열에 넣는다. 하나의 점(point)에 필요한 x, y, z를 한 객체에 담고, 그 객체들을 배열로 나열한다.

```typescript
const points = [];
for (let i = 0; i < N; i++) {
  points.push({ x: Math.random(), y: Math.random(), z: Math.random() });
}
```

일상적인 비유로 보면, **서랍장에 사람별로 서류를 보관하는 것**과 같다. "홍길동 서랍"을 열면 이력서, 계약서, 급여명세서가 한 곳에 있다.

### SOA (Structure of Arrays) — 배열 구조체

데이터를 **"속성 단위"**로 분리해서, 같은 종류끼리 모아둔다. x값은 x끼리, y값은 y끼리, z값은 z끼리 별도의 배열에 저장한다.

```typescript
const points = {
  x: new Float64Array(N),
  y: new Float64Array(N),
  z: new Float64Array(N),
};
for (let i = 0; i < N; i++) {
  points.x[i] = Math.random();
  points.y[i] = Math.random();
  points.z[i] = Math.random();
}
```

일상적인 비유로 보면, **종류별로 파일 캐비닛을 나누는 것**과 같다. "이력서 캐비닛", "계약서 캐비닛", "급여명세서 캐비닛"이 따로 있고, 각 캐비닛 안에서 사람 순서대로 정렬되어 있다.

## 2. 성능 테스트

100만 개의 3D 좌표를 생성하고, 모든 좌표의 x + y + z 합을 순회하며 계산하는 동일한 작업을 두 패턴으로 수행했다.

### 테스트 코드

```typescript
describe('AOS vs SOA', () => {
  const N = 1_000_000;

  it('aos', async () => {
    const points = [];
    for (let i = 0; i < N; i++) {
      points.push({ x: Math.random(), y: Math.random(), z: Math.random() });
    }

    let sum = 0;
    for (let i = 0; i < N; i++) {
      sum += points[i].x + points[i].y + points[i].z;
    }
    console.log(sum);
  });

  it('soa', async () => {
    const points = {
      x: new Float64Array(N),
      y: new Float64Array(N),
      z: new Float64Array(N),
    };
    for (let i = 0; i < N; i++) {
      points.x[i] = Math.random();
      points.y[i] = Math.random();
      points.z[i] = Math.random();
    }

    let sum = 0;
    for (let i = 0; i < N; i++) {
      sum += points.x[i] + points.y[i] + points.z[i];
    }
    console.log(sum);
  });
});
```

### 결과

| 패턴 | 실행 시간 | 배수 |
|------|----------|------|
| AOS (객체 배열) | **104ms** | 기준 |
| SOA (배열 구조체) | **24ms** | **약 4.3배 빠름** |

동일한 100만 건의 데이터를 동일하게 순회했는데, 데이터를 배치하는 방식만 다를 뿐인데 4배 이상의 성능 차이가 발생했다.

## 3. 왜 이런 차이가 나는가

### 3.1. 컴퓨터가 데이터를 읽는 방식 — 캐시 라인

성능 차이의 가장 큰 원인은 **CPU 캐시**와 관련이 있다.

CPU가 메모리에서 값을 읽는 데는 시간이 걸린다. CPU 내부 연산은 약 0.3나노초가 걸리지만, 메모리에서 데이터를 가져오는 데는 약 100나노초가 걸린다. **약 300배의 속도 차이**가 있는 것이다.

이 차이를 줄이기 위해 CPU에는 **캐시**라는 작은 고속 메모리가 있다. 비유하자면 이렇다:

> CPU는 **책상에서 일하는 사람**이고, 메모리는 복도 끝에 있는 **창고**이다.
> 책상 위의 자료는 바로 쓸 수 있지만, 창고까지 다녀오려면 한참 걸린다.

여기서 중요한 점은 창고에 갈 때 **요청한 서류 1장만 가져오지 않는다**는 것이다. 그 서류가 들어 있는 **서류철 전체**(보통 64바이트)를 통째로 가져온다. 이것이 바로 **캐시 라인**이다.

만약 다음에 필요한 서류가 방금 가져온 서류철 안에 있으면, 창고에 다시 갈 필요가 없다. 이것을 **캐시 히트**라고 부른다. 반대로, 필요한 자료가 서류철에 없어서 창고에 다시 가야 하는 상황을 **캐시 미스**라고 부른다.

### 3.2. AOS — 매번 창고를 다녀와야 한다

AOS에서 `{ x, y, z }` 객체를 만들면, `points` 배열에는 실제 데이터가 아니라 각 객체가 어디 있는지를 가리키는 **주소(포인터)**만 들어 있다.

> 아래 그림은 이해를 돕기 위한 **비유적 표현**이다. 실제로 V8은 객체를 연속 생성하면 Young Generation 영역에서 **인접하게 배치**한다. 하지만 객체 사이에 헤더와 메타데이터가 끼어들어 **순수 데이터의 연속성이 깨지고**, GC가 발생하면 Old Generation으로 이동하면서 **실제로 흩어질 수 있다.** 또한 매번 포인터를 따라가야 하는 간접 참조(indirection)가 필요하므로 캐시 효율이 떨어진다.

```
points 배열:  [주소0] [주소1] [주소2] [주소3] ...

주소0 → 메모리 어딘가에 { x: 0.42, y: 0.87, z: 0.13 }
주소1 → 메모리 다른 곳에 { x: 0.55, y: 0.31, z: 0.92 }
주소2 → 메모리 또 다른 곳에 { x: 0.71, y: 0.64, z: 0.28 }
```

순회할 때 벌어지는 일:

```
1. points[0]의 주소를 읽음     → 창고 방문 1회
2. 그 주소로 가서 x, y, z를 읽음 → 창고 방문 2회
3. points[1]의 주소를 읽음     → 창고 방문 3회
4. 그 주소로 가서 x, y, z를 읽음 → 창고 방문 4회
...
```

각 객체가 메모리에 흩어져 있기 때문에, 서류철을 가져와도 **다음에 필요한 객체가 그 안에 없다.** 매번 창고를 왕복해야 하니 느릴 수밖에 없다.

### 3.3. SOA — 한 번 다녀오면 여러 번 쓸 수 있다

SOA에서 `Float64Array`는 숫자를 **빈틈없이 연속으로** 저장하는 특수 배열이다. 일반 객체처럼 여기저기 흩어지지 않는다.

```
x 배열: [x0][x1][x2][x3][x4][x5][x6][x7][x8]...  ← 메모리에 나란히 붙어 있음
y 배열: [y0][y1][y2][y3][y4][y5][y6][y7][y8]...
z 배열: [z0][z1][z2][z3][z4][z5][z6][z7][z8]...
```

`Float64Array`의 원소 하나는 정확히 8바이트이다. 캐시 라인(64바이트)에는 8개의 원소가 들어간다.

```
1. x[0]을 읽음 → 창고 방문 1회 (x[0] ~ x[7]이 서류철에 함께 온다)
2. x[1]을 읽음 → 이미 책상 위에 있음 (창고 방문 불필요)
3. x[2]을 읽음 → 이미 책상 위에 있음
...
8. x[7]을 읽음 → 이미 책상 위에 있음
9. x[8]을 읽음 → 창고 방문 2회 (x[8] ~ x[15]이 함께 온다)
```

**8번 접근 중 7번은 창고에 가지 않아도 된다.** 이것이 연속 메모리 배치의 힘이다.

### 3.4. 메모리 사용량 차이

JavaScript에서 일반 객체를 만들면, 값 자체 외에도 엔진이 관리하는 **부가 정보**가 따라붙는다. 이 객체가 어떤 형태인지(hidden class), 가비지 컬렉터가 추적하기 위한 정보 등이 포함된다.

| | AOS (객체) | SOA (Float64Array) |
|---|---|---|
| x, y, z 값 자체 | 24 bytes | 24 bytes |
| 부가 정보 (객체 헤더, 타입 정보, GC 메타데이터) | ~40-56 bytes 추가 | 없음 |
| **요소 1개당 합계** | **~64-80 bytes** | **24 bytes** |
| **100만 개 총합** | **~64-80 MB** | **~24 MB** |

`Float64Array`는 C 언어의 배열처럼 숫자 값만 빈틈없이 저장하기 때문에 부가 정보가 없다. 같은 데이터인데 메모리를 3분의 1만 사용한다.

메모리를 적게 쓴다는 것은 캐시에 더 많은 데이터를 담을 수 있다는 뜻이기도 하다. 이것이 캐시 히트율을 더욱 높인다.

### 3.5. 가비지 컬렉션(GC) 부담 차이

JavaScript 엔진은 주기적으로 **더 이상 사용하지 않는 객체를 찾아서 메모리를 회수**하는 작업을 수행한다. 이것을 가비지 컬렉션(GC)이라고 부른다.

- **AOS**: 100만 개의 개별 객체가 메모리에 존재한다. GC는 이 100만 개를 하나하나 "아직 사용 중인가?"를 확인해야 한다.
- **SOA**: GC가 확인할 객체는 **`Float64Array` 3개**뿐이다. 배열 내부의 숫자들은 GC의 추적 대상이 아니다.

GC가 동작하는 동안에는 프로그램 실행이 잠시 멈출 수 있는데, 추적할 객체가 100만 개인 경우와 3개인 경우의 부담은 비교할 수 없을 정도로 다르다.

## 4. 정리

| 비교 항목 | AOS (객체 배열) | SOA (배열 구조체) |
|----------|----------------|------------------|
| 데이터 배치 | 메모리에 흩어짐 | 나란히 연속 배치 |
| 캐시 효율 | 매번 창고 방문 (캐시 미스 빈번) | 8번 중 1번만 방문 (캐시 히트) |
| 메모리 사용 | ~64-80 MB | ~24 MB |
| GC 부담 | 100만 개 객체 추적 | 3개 객체 추적 |
| **실행 시간** | **104ms** | **24ms** |

핵심은 **"다음에 필요한 데이터가 이미 가까이에 있느냐"**이다.

AOS는 직관적이고 코드가 읽기 쉽다. 대부분의 비즈니스 로직에서는 AOS로 충분하다. 하지만 대량의 데이터를 순차적으로 처리하는 성능 민감 구간에서는 SOA + `TypedArray` 조합이 압도적으로 유리하다.

이 패턴은 게임 엔진의 물리 연산, ECS(Entity Component System) 아키텍처, 대규모 데이터 시각화, 과학 시뮬레이션 등에서 널리 활용되고 있다.
