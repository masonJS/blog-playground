## 1. io_uring이란 무엇인가

### 배경: epoll 이후의 질문

[이전 글](./11.%20입출력%20멀티플렉싱.md)에서 `select → poll → epoll`로 이어지는 I/O 멀티플렉싱의 진화를 살펴봤다. epoll은 **readiness(준비 상태) 통지** 모델로, "이 소켓에 데이터가 도착했으니 네가 읽어"라고 알려주는 방식이다.

그런데 이 모델에는 본질적인 한계가 있다. 커널이 "준비됐어"라고 알려준 뒤에도 **애플리케이션이 직접 `read()` 시스템 콜을 호출**해야 한다. 통지와 실제 I/O가 분리되어 있다.

2019년, Linux 5.1에서 이 구조 자체를 바꾸는 새로운 인터페이스가 등장했다. **io_uring**이다.

### 핵심 아이디어: 시스템 콜 없이 I/O를 주고받자

기존의 모든 I/O 인터페이스는 시스템 콜을 경유해야 했다. `read()`, `write()`, `epoll_wait()` 전부 유저↔커널 경계를 넘는 비용을 수반한다.

io_uring은 **유저 공간과 커널 공간이 공유하는 메모리 영역(링 버퍼)**을 만들어서, 시스템 콜 없이도 I/O 요청과 완료 결과를 주고받을 수 있게 했다.

### 두 개의 링 버퍼: SQ와 CQ

io_uring의 구조는 두 개의 링 버퍼로 이루어진다.

- **Submission Queue (SQ)**: 애플리케이션이 I/O 요청을 넣는 큐
- **Completion Queue (CQ)**: 커널이 완료된 결과를 넣는 큐

```
┌─────────────────────────────────────────────────────────┐
│                    공유 메모리 영역                       │
│                                                         │
│   Submission Queue (SQ)         Completion Queue (CQ)   │
│  ┌───┬───┬───┬───┬───┐       ┌───┬───┬───┬───┬───┐    │
│  │ ● │ ● │   │   │   │       │ ✓ │   │   │   │   │    │
│  └───┴───┴───┴───┴───┘       └───┴───┴───┴───┴───┘    │
│    ↑                            ↑                       │
│    앱이 요청을 넣음               커널이 결과를 넣음       │
│                                                         │
└─────────────────────────────────────────────────────────┘
        ↑ 앱이 접근                   ↑ 커널이 접근
        ↑ 커널이 접근                  ↑ 앱이 접근
```

두 큐 모두 유저 공간과 커널 공간에서 동시에 접근할 수 있다. 이 **공유 메모리 구조** 덕분에 매번 시스템 콜로 데이터를 복사하지 않아도 된다.

### 동작 흐름

```
  애플리케이션                                커널
 ┌──────────┐                           ┌──────────┐
 │          │                           │          │
 │ 1. SQE 작성                           │          │
 │    "fd7을 읽어줘"                      │          │
 │          │                           │          │
 │ 2. SQ tail 포인터 업데이트               │          │
 │    (시스템 콜 없이 메모리 쓰기만)           │          │
 │          │                           │          │
 │ 3. io_uring_enter()  ──────────►     │ 4. SQ에서 요청 꺼냄    
 │    "처리할 게 있어"                     │    fd7 read 수행       
 │    (배치로 한 번만)                     │                      
 │          │                           │ 5. 완료!               
 │          │                           │    CQE를 CQ에 넣음     
 │          │                           │    "fd7 읽기 완료,      
 │          │                           │     2048바이트"     
 │          │                           │          │
 │ 6. CQ에서 결과 꺼냄   ◄──────────       │          │
 │    (시스템 콜 없이 메모리 읽기만)           │          │
 │          │                           │          │
 └──────────┘                           └──────────┘
```

핵심은 3단계의 `io_uring_enter()`이다. 이 한 번의 시스템 콜로 **SQ에 쌓인 여러 요청을 한꺼번에 제출**할 수 있다. 그리고 6단계에서 결과를 가져올 때는 시스템 콜이 아예 필요 없다. CQ 메모리를 직접 읽으면 된다.

더 나아가 **SQPOLL 모드**를 사용하면 커널 스레드가 SQ를 주기적으로 폴링하므로, `io_uring_enter()` 호출마저 생략할 수 있다. 이 경우 **시스템 콜 0번**으로 I/O가 가능해진다.

### Readiness vs Completion 모델

epoll과 io_uring의 가장 근본적인 차이는 **통지 모델**이다.

```
epoll (Readiness 모델):
  앱: "fd7 감시해줘"
  커널: "fd7 읽을 준비 됐어!"          ← 통지
  앱: read(fd7, buffer, size)         ← 앱이 직접 I/O 수행
  커널: (데이터를 버퍼에 복사)

io_uring (Completion 모델):
  앱: "fd7을 읽어서 이 버퍼에 넣어줘"   ← 요청 (SQ에 넣음)
  커널: (데이터를 읽어서 버퍼에 복사)     ← 커널이 I/O 수행
  커널: "fd7 읽기 완료, 2048바이트"     ← 완료 통지 (CQ에 넣음)
```

비유하자면:

- **epoll**: 택배 기사가 "짐이 1층에 도착했어요" 문자를 보낸다. 직접 내려가서 가져와야 한다.
- **io_uring**: 택배 기사가 짐을 문 앞까지 갖다 놓고 "문 앞에 뒀어요" 문자를 보낸다.

### C 의사 코드로 보는 io_uring

```c
// io_uring 초기화
struct io_uring ring;
io_uring_queue_init(256, &ring, 0);  // 256개 엔트리 크기의 링 생성

// I/O 요청 제출
struct io_uring_sqe *sqe = io_uring_get_sqe(&ring);  // SQ 엔트리 하나 확보
io_uring_prep_read(sqe, fd, buffer, size, 0);         // "이 fd를 읽어줘"
io_uring_submit(&ring);                                // 커널에 제출

// 완료 대기
struct io_uring_cqe *cqe;
io_uring_wait_cqe(&ring, &cqe);     // CQ에서 완료 이벤트 꺼냄
int bytes_read = cqe->res;           // 읽은 바이트 수
io_uring_cqe_seen(&ring, cqe);       // 처리 완료 표시
```

epoll에서는 `epoll_wait()` → `read()` 두 단계가 필요했지만, io_uring에서는 **요청 제출과 결과 수신만** 있으면 된다. `read()` 시스템 콜을 직접 호출하지 않는다.

## 2. 기존 libuv의 epoll과 비교

### libuv의 이중 구조: epoll + 스레드 풀

[이전 글](./11.%20입출력%20멀티플렉싱.md)에서 Node.js가 libuv를 통해 I/O 멀티플렉싱을 수행한다고 설명했다. 그런데 libuv의 실제 아키텍처를 들여다보면, **모든 I/O가 epoll로 처리되는 것은 아니다**.

```
┌──────────────────────────────────────────────────┐
│                   libuv                          │
│                                                  │
│   네트워크 I/O          파일 I/O                    │
│  ┌──────────────┐    ┌──────────────────────┐    │
│  │    epoll     │    │     스레드 풀          │    │
│  │  (이벤트 루프)  │    │  (기본 4개 스레드)      │    │
│  │              │    │                      │    │
│  │  소켓 read    │    │  fs.readFile()  ──► 워커 스레드 1  
│  │  소켓 write   │    │  fs.stat()     ──► 워커 스레드 2  
│  │  TCP accept  │    │  fs.writeFile() ──► 워커 스레드 3  
│  │  DNS(일부)    │    │  DNS lookup    ──► 워커 스레드 4  
│  └──────────────┘    └──────────────────────┘    │
└──────────────────────────────────────────────────┘
```

**네트워크 소켓**은 epoll로 효율적으로 감시할 수 있다. 소켓 버퍼에 데이터가 도착하면 커널이 이벤트를 발생시키니까.

하지만 **일반 파일(디스크 I/O)**은 다르다. Linux에서 일반 파일의 fd는 epoll에 등록할 수 없다. 일반 파일은 항상 "ready" 상태로 간주되기 때문이다. 실제로는 디스크에서 데이터를 읽어오는 데 시간이 걸리지만, 커널의 관점에서 파일 fd는 즉시 읽을 수 있는 것으로 취급된다.

그래서 libuv는 파일 I/O를 **별도의 스레드 풀**에서 처리한다. `fs.readFile()`을 호출하면 워커 스레드가 블로킹 `read()`를 수행하고, 완료되면 이벤트 루프에 통지한다.

### 스레드 풀의 한계

이 방식은 잘 동작하지만, 병목이 될 수 있다.

```
  요청 급증 시:

  fs.readFile(A)  ──► 워커 1 (처리 중...)
  fs.readFile(B)  ──► 워커 2 (처리 중...)
  fs.readFile(C)  ──► 워커 3 (처리 중...)
  fs.readFile(D)  ──► 워커 4 (처리 중...)
  fs.readFile(E)  ──► ⏳ 대기열... (스레드 풀이 가득 참)
  fs.readFile(F)  ──► ⏳ 대기열...
  fs.readFile(G)  ──► ⏳ 대기열...
  dns.lookup(H)   ──► ⏳ 대기열... (DNS도 스레드 풀을 공유!)
```

- 기본 스레드 풀 크기는 **4개**이다 (`UV_THREADPOOL_SIZE` 환경변수로 조정 가능, 최대 1024)
- 파일 I/O와 DNS lookup이 같은 스레드 풀을 공유한다
- 대량의 파일 I/O가 발생하면 DNS 조회까지 밀릴 수 있다
- 스레드마다 컨텍스트 스위칭 비용이 발생한다

### io_uring이 해결하려 한 것

io_uring은 이 구조를 근본적으로 바꿀 수 있었다.

```
기존 (epoll + 스레드 풀):
  fs.readFile()
      │
      ▼
  스레드 풀에 작업 제출
      │
      ▼
  워커 스레드가 블로킹 read() 수행    ← 스레드 필요
      │
      ▼
  완료 → 이벤트 루프에 통지

io_uring 적용 시:
  fs.readFile()
      │
      ▼
  SQ에 read 요청 제출               ← 스레드 불필요!
      │
      ▼
  커널이 비동기로 read 수행
      │
      ▼
  CQ에 결과 도착 → 이벤트 루프에 통지
```

스레드 풀 없이도 파일 I/O를 비동기로 처리할 수 있게 되는 것이다.

### 성능 비교

libuv에 io_uring이 도입되었을 때, [Phoronix 벤치마크](https://www.phoronix.com/news/libuv-io-uring)에서 측정한 결과:

| 벤치마크 항목 | epoll + 스레드 풀 | io_uring | 향상 |
|---|---|---|---|
| 파일 I/O 처리량 | 기준 | **최대 ~8배** | 스레드 풀 오버헤드 제거 |
| 시스템 콜 횟수 | read() 호출 × N | 배치 제출 1회 | 시스템 콜 비용 절감 |
| 컨텍스트 스위칭 | 워커 스레드 전환 발생 | 단일 스레드 처리 가능 | 전환 비용 제거 |

### 종합 비교

| 비교 항목 | epoll + 스레드 풀 (기존) | io_uring |
|---|---|---|
| 네트워크 I/O | epoll로 이벤트 감시 | io_uring으로 통합 가능 |
| 파일 I/O | 스레드 풀에서 블로킹 처리 | 커널이 비동기 처리 |
| 통지 모델 | readiness (준비 알림) | completion (완료 알림) |
| 스레드 필요 여부 | 파일 I/O에 워커 스레드 필요 | 불필요 |
| 시스템 콜 | 작업당 개별 호출 | 배치 제출 가능 |
| 배치 처리 | 불가 | SQ에 여러 요청 한꺼번에 |
| 커널 요구 사항 | Linux 2.6+ | Linux 5.1+ |

## 3. libuv의 실험적 도입과 철회

### 타임라인

```
2019.05  Linux 5.1 릴리즈, io_uring 도입
    │
    ▼
2023.05  libuv v1.45.0 — io_uring 지원 추가, 기본 활성화
    │    (read, write, fsync, stat 등 파일 I/O 대상)
    │
    ▼
2024.02.13  Node.js — io_uring 기본값 비활성화 (opt-out → opt-in)
    │
    ▼
2024.02.14  Node.js 보안 릴리즈 — CVE-2024-22017 공개
    │
    ▼
2024.09  libuv v1.49.0 — SQPOLL io_uring 기본 비활성화
    │
    ▼
현재    기본 경로는 스레드 풀 + 일부 io_uring 최적화 병행
```

### libuv v1.45.0: 도입

2023년 5월, [libuv v1.45.0에서 io_uring 지원이 추가](https://github.com/libuv/libuv/commit/d2c31f429b87b476a7f1344d145dad4752a406d4)되었다. 대상 오퍼레이션은 `read`, `write`, `fsync`, `fdatasync`, `stat`, `fstat`, `lstat` 등 파일 I/O였다. 커널이 io_uring을 지원하지 않는 환경에서는 자동으로 기존 스레드 풀로 폴백했다.

이 시점에서 io_uring은 **기본 활성화** 상태였다. 비활성화하려면 환경변수 `UV_USE_IO_URING=0`을 설정해야 했다.

### CVE-2024-22017: setuid() 권한 강등 실패

2024년 2월, Node.js 보안 릴리즈에서 [CVE-2024-22017](https://nodejs.org/en/blog/vulnerability/february-2024-security-releases/)이 공개되었다.

**문제의 핵심**: 프로세스가 `setuid()`로 권한을 낮추기 전에 io_uring이 이미 초기화되어 있으면, io_uring 링 버퍼가 **높은 권한 상태에서 생성된 커널 컨텍스트를 유지**한다. `setuid()` 이후에도 io_uring을 통해 제출된 I/O 작업은 **이전의 높은 권한으로 실행**될 수 있었다.

```
일반적인 기대:
  root로 시작 → setuid(1000) → 이후 모든 작업은 uid 1000 권한

실제 발생한 일:
  root로 시작 → io_uring 초기화 (root 컨텍스트) → setuid(1000)
  → io_uring을 통한 I/O는 여전히 root 권한으로 실행!
```

이것은 io_uring의 **설계적 특성**에서 비롯된 문제였다. io_uring의 커널 워커 스레드는 링 생성 시점의 credential을 캡처하는데, 이후 프로세스가 `setuid()`로 credential을 변경해도 이미 생성된 커널 워커에는 반영되지 않았다.

영향받은 버전: Node.js v20.x, v21.x (Node 보안 공지 기준)

관련 이슈: [libuv #4416](https://github.com/libuv/libuv/issues/4416), [Node.js #52156](https://github.com/nodejs/node/issues/52156)

### Node.js의 대응: 기본값 비활성화

CVE 공개 하루 전인 2024년 2월 13일, [Node.js는 io_uring을 기본값에서 비활성화](https://github.com/nodejs/node/commit/686da19abb)했다.

```
변경 전 (libuv v1.45.0 ~ v1.48.x):
  io_uring 경로 기본 활성화
  비활성화하려면: UV_USE_IO_URING=0

변경 후 (libuv v1.49.0+):
  SQPOLL io_uring 기본 비활성화
  SQPOLL을 쓰려면: UV_USE_IO_URING=1
```

### 더 큰 그림: 커널 레벨의 io_uring 보안 문제

Node.js/libuv의 CVE는 빙산의 일각이었다. io_uring 자체가 Linux 커널에서 **보안 취약점의 온상**이 되고 있었다.

- **CVE-2021-41073**: io_uring의 타입 혼동으로 인한 로컬 권한 상승
- **CVE-2023-2598**: io_uring의 메모리 관리 결함으로 커널 메모리 접근 가능
- **CVE-2024-0582**: io_uring의 use-after-free 취약점

Google은 이 문제를 가장 적극적으로 대응한 기업 중 하나였다.

> "io_uring은 성능상의 이점을 가져다주지만… 여전히 심각한 취약점의 영향을 받고 있으며, 강력한 익스플로잇 프리미티브를 제공한다. 이러한 이유로 현재는 신뢰할 수 있는 컴포넌트에서만 사용하는 것이 안전하다고 판단한다."
> — [Google Security Blog](https://www.phoronix.com/news/Google-Restricting-IO_uring)

구체적인 조치:

| 환경 | 조치 |
|---|---|
| Chrome OS | io_uring 완전 비활성화 |
| Android | seccomp-bpf 필터로 앱의 io_uring 접근 차단 |
| GKE AutoPilot | io_uring 기본 비활성화 |
| Google 프로덕션 서버 | 인프라 전체에서 io_uring 비활성화 |

Google의 2022년 취약점 보상 프로그램(VRP) 보고의 **60%가 io_uring 관련**이었으며, io_uring 취약점에 대해 약 **100만 달러**의 바운티가 지급되었다.

### 현재 상태

2026년 현재, libuv의 최신 버전(v1.52.1)에서 io_uring 코드는 여전히 유지되고 있다. 기본 경로는 스레드 풀을 사용하지만, 일부 io_uring 최적화 경로는 기본 동작에 포함될 수 있다. `UV_USE_IO_URING=1`은 주로 SQPOLL 같은 io_uring 경로를 명시적으로 활성화할 때 사용한다.

```bash
# io_uring 활성화 (opt-in)
UV_USE_IO_URING=1 node app.js
```

```
현재 libuv 아키텍처 (요약):

┌──────────────────────────────────────────┐
│                 libuv                    │
│                                          │
│   네트워크 I/O        파일 I/O           │
│  ┌──────────┐     ┌──────────────┐      │
│  │  epoll   │     │  스레드 풀    │      │
│  │  kqueue  │     │  (기본 경로)  │      │
│  │  IOCP    │     │              │      │
│  └──────────┘     └──────────────┘      │
│                                          │
│   일부 io_uring 최적화는 기본 경로에 포함 가능 │
│   UV_USE_IO_URING=1: SQPOLL 등 명시 활성화     │
└──────────────────────────────────────────┘
```

io_uring의 성능 이점은 분명하지만, 커널 보안 문제가 충분히 해결되기 전까지는 프로덕션 환경에서의 기본 활성화는 당분간 어려울 것으로 보인다. 성능과 보안 사이의 트레이드오프에서, Node.js 생태계는 **보안을 선택**했다.
