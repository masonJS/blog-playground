# AWS Agentic AI 세션 정리

> AWS Bedrock, Strands Agent SDK, AgentCore를 활용한 생성형 AI 에이전트 구축 가이드

---

## 1. AWS Bedrock 기반 RAG 서비스 구축

### 1.1 청킹(Chunking) 전략

RAG 서비스를 구축할 때 가장 먼저 마주하는 과제는 문서를 임베딩 모델이 잘 이해할 수 있는 올바른 단위로 분할하는 것이다. 이를 **청킹(Chunking)**이라 부르며, 청킹 전략에 따라 검색 품질이 크게 달라진다.

청킹 크기의 트레이드오프는 명확하다. 너무 크게 자르면 검색 정확도가 떨어지고, 너무 작게 자르면 맥락이 사라진다. 대표적인 청킹 방식으로는 고정 크기(Fixed-size) 청킹, 문장 단위 청킹, 의미 단위(Semantic) 청킹 등이 있으며, 문서의 특성과 서비스 요구사항에 맞는 전략을 선택해야 한다.

### 1.2 임베딩 모델 선택

임베딩 모델은 문서와 질의를 벡터 공간에 매핑하는 핵심 구성요소다. 특히 한국어 서비스를 운영한다면 한국어 지원이 잘 되는 임베딩 모델을 선택하는 것이 중요하다. 한국어 특유의 교착어적 특성, 조사 처리, 띄어쓰기 변형 등을 잘 처리할 수 있는 모델이어야 검색 품질을 보장할 수 있다.

### 1.3 벡터 데이터베이스 관리

벡터 데이터베이스의 선택은 서비스 규모와 문서 특성에 따라 달라진다. 문서가 수천 건일 때와 백만 건일 때 필요한 데이터베이스가 다르고, 문서의 타입이나 복잡도에 따라서도 적합한 벡터 데이터베이스가 달라질 수 있다. 규모가 작을 때는 경량 솔루션으로 충분하지만, 대규모 서비스에서는 분산 처리와 인덱싱 성능이 중요해진다.

### 1.4 검색 정확도 향상 기법

기본적인 벡터 유사도 검색 외에도 검색 정확도를 높이는 다양한 기법들이 존재한다. 리랭킹(Re-ranking), 하이브리드 검색(키워드 + 시맨틱), 쿼리 확장 등의 기법을 적절히 활용하면 검색 품질을 크게 개선할 수 있다. 다만 이 모든 것을 직접 구현하기는 어렵기 때문에, AWS Bedrock과 같은 관리형 서비스를 활용하는 것이 효율적이다.

## 2. 프롬프트 캐싱(Prompt Caching)

LLM을 반복적으로 호출하면 토큰 비용이 크게 증가한다. 이 문제를 해결하기 위해 **프롬프트 캐싱** 기능이 제공된다.

프롬프트 캐싱은 네트워크 체크포인팅과 유사한 개념이다. 프롬프트의 특정 지점을 체크포인트로 지정하면, 해당 지점까지의 연산은 다시 수행하지 않아도 된다. 이를 통해 LLM 호출 시 캐싱으로 지정된 부분은 재연산되지 않기 때문에 훨씬 빠른 속도로 답변을 생성할 수 있고, 비용도 그만큼 절감할 수 있다.

## 3. AWS Bedrock 보안 및 컴플라이언스

### 3.1 데이터 보호

고객들이 Bedrock 서비스를 검토할 때 가장 먼저 묻는 질문은 "우리가 입력한 데이터는 어떻게 되는가"이다. AWS Bedrock은 입력 데이터와 모델이 생성한 출력 데이터 모두 파운데이션 모델의 학습에 사용되지 않는다. 이는 기업 고객의 데이터 주권과 기밀성을 보장하는 핵심 원칙이다.

### 3.2 네트워크 보안

기본적으로 모든 통신은 TLS 암호화로 보호된다. 더 강한 보안 요구사항이 있는 경우, **AWS PrivateLink**를 통해 트래픽이 인터넷을 전혀 거치지 않도록 하는 옵션도 제공한다. 이를 통해 VPC 내부에서 직접 Bedrock 엔드포인트에 접근할 수 있다.

### 3.3 컴플라이언스 인증

AWS Bedrock은 GDPR, SOC, ISO, HIPAA 등 주요 규제 프레임워크를 이미 충족하고 있어, 금융이나 헬스케어 같은 규제 산업에서도 도입이 가능하다.

## 4. 가드레일(Guardrails)

프롬프트 내에서 특정 응답을 제한하는 것도 가능하지만, **가드레일** 기능은 한 단계 더 나아가 시스템 수준에서 필터링을 수행한다.

가드레일이 필터링할 수 있는 대표적인 항목은 다음과 같다.

- **비즈니스 정책 위반 방지**: 예를 들어 "청소년은 절대 안 된다"는 연령 제한 정책이나, "환불이 된다는 얘기를 하면 안 된다"는 CS 정책 등을 에이전트가 위반하지 않도록 차단한다.
- **유해 콘텐츠 필터링**: 혐오 표현, 폭력적 콘텐츠, 성적 콘텐츠 등 유해한 내용을 자동으로 감지하고 차단한다.

프롬프트 수준의 지시만으로는 LLM이 이를 100% 준수한다고 보장하기 어렵기 때문에, 가드레일을 함께 활용하여 이중 안전장치를 구축하는 것이 권장된다.

## 5. Strands Agent SDK

### 5.1 개요와 배경

Strands Agent SDK는 매우 짧은 코드로 AI 에이전트를 구현할 수 있게 해주는 오픈소스 Python 프레임워크다. 기존에는 LangGraph와 같이 복잡한 에이전트 프레임워크가 주류였으나, LLM의 리즈닝(Reasoning) 능력이 크게 향상되면서 패러다임이 변화했다.

### 5.2 LangGraph와의 비교

LangGraph 같은 프레임워크는 에이전트 개발 시 그래프 방식으로 인풋과 아웃풋을 명확하게 정의하고, 각 노드 간의 흐름을 세밀하게 제어해야 했다. 많은 사용자가 사용했지만, 그만큼 무겁고 번거로운 측면이 있었다.

반면 Strands Agent SDK는 리즈닝 모델의 발전에 기반한다. 최신 LLM은 프롬프트를 읽으면 어떤 액션을 취해야 하는지, 어떤 도구를 호출해야 하는지를 스스로 잘 판단할 수 있다. 따라서 개발자가 모든 흐름을 명시적으로 정의할 필요 없이, 간결한 코드만으로 강력한 에이전트를 만들 수 있다.

### 5.3 에이전트 루프(Agent Loop)

AI 에이전트의 핵심 동작 원리는 **에이전트 루프**라 불리는 반복적 추론-실행 사이클이다. 그 동작 과정은 다음과 같다.

1. **입력 이해**: 사용자의 요청(인풋)이 들어오면 LLM이 이를 먼저 이해한다.
2. **태스크 분해**: 요청을 수행하기 위해 필요한 태스크들을 나누어 정리한다.
3. **도구 실행**: 정리된 태스크를 수행하기 위해 필요한 도구들을 선택하고 실행한다.
4. **결과 평가**: 도구 호출의 결과를 바탕으로 추가 리즈닝이 필요한지 판단한다.
5. **반복 또는 완료**: 결과가 충분하지 않으면 다시 도구를 호출하고, 충분하면 최종 응답을 생성한다.

이 루프를 통해 에이전트는 복잡한 다단계 작업도 자율적으로 수행할 수 있다.

### 5.4 도구(Tool) 사용 방식

Strands Agent SDK에서 도구를 사용하는 방식은 크게 세 가지로 나뉜다.

#### 커스텀 툴 (Custom Tools)

Python의 `@tool` 데코레이터를 사용하여 함수를 도구로 등록한다. 데코레이터 위에 작성한 docstring(주석)을 LLM이 읽고, 해당 도구를 언제 어떻게 사용할지 자동으로 판단한다.

```python
@tool
def search_database(query: str) -> str:
    """데이터베이스에서 고객 정보를 검색합니다."""
    # 구현 로직
```

#### MCP 툴 (Model Context Protocol)

MCP는 에이전트와 외부 도구 간의 표준화된 통신 프로토콜이다. 세 가지 연결 방식을 제공한다.

- **STDIO**: 로컬 환경에서 사용하며, 지연 시간이 매우 짧다. CLI 도구나 내부 도구를 만들 때 주로 활용된다. 애플리케이션을 서비스로 올리는 방식은 아니지만, 로컬 통신 기반이므로 레이턴시가 최소화된다.
- **Streamable HTTP**: MCP 서버가 원격에 있는 경우 많이 사용되는 방식이다. `MCPClient`에 URL을 입력하여 HTTP 요청 기반으로 통신한다. 가장 범용적으로 사용되는 방식이다.
- **SSE (Server-Sent Events)**: Streamable HTTP와 유사하며, 레거시 호환 목적으로 제공된다.

대부분의 경우 직접 MCP 서버를 만들기보다는 오픈소스로 공개된 MCP 도구를 찾아서 에이전트에 연결하는 방식으로 활용한다.

#### 빌트인 툴 (Built-in Tools)

SDK에 기본적으로 내장된 도구들로, 별도의 설정 없이 바로 사용할 수 있다.

### 5.5 멀티 에이전트 패턴

복잡한 작업을 처리하기 위해 여러 에이전트를 조합하는 멀티 에이전트 방식은 네 가지 대표적인 패턴이 있다.

#### Agent as Tool (에이전트를 도구로 사용)

상위 에이전트가 전문화된 하위 에이전트에게 작업을 위임하고, 결과를 통합하는 계층적 구조다. 오케스트레이터 에이전트가 전체 흐름을 관리하면서, 특정 작업은 전문 에이전트에게 맡기는 방식이다.

#### Swarm (스웜)

동등한 위치의 에이전트들이 각자 고유한 전문 분야(스페셜티)를 가지고 협업하는 구조다. 예를 들어 코드를 작성하는 에이전트, 리서치를 수행하는 에이전트, 답변을 비교하고 피드백을 주는 에이전트, 이를 종합하여 아키텍처를 설계하는 에이전트가 각각 자신의 전문 영역에서 기여한다.

#### Graph (그래프)

에이전트 간 정보 흐름을 특정 방향성을 가진 네트워크로 명시적으로 정의하는 방식이다. 예를 들어 고객 문의가 들어오면 카테고리를 분류하는 에이전트가 먼저 동작하고, 분류 결과에 따라 환불 처리 에이전트, 기술 지원 에이전트 등 해당 담당 에이전트로 라우팅된다.

#### 기타 패턴

위의 세 가지 외에도 다양한 조합이 가능하며, 서비스 요구사항에 맞게 패턴을 선택하거나 혼합하여 사용할 수 있다.

## 6. POC에서 프로덕션으로: 엔터프라이즈 챌린지

POC(Proof of Concept)에서 좋은 결과를 얻었더라도, 실제 서비스에 올리려면 다양한 챌린지에 직면하게 된다. 주요 과제는 다음과 같다.

### 6.1 세션 격리

고객 A의 상담 내용이 고객 B에게 노출되면 안 된다. 세션이 완전히 격리되어야 하며, 이는 생각보다 구현하기 어려운 요건이다.

### 6.2 메모리 관리

에이전트는 단기 기억(현재 대화)뿐만 아니라 장기 기억(과거 대화, 주문 이력 등)도 유지해야 한다. 사용자가 지난주에 무엇을 주문했는지, 이전에 어떤 질문을 했는지 등을 기억하고 있어야 자연스러운 서비스가 가능하다.

### 6.3 외부 시스템 연동

에이전트가 실제 업무를 수행하려면 CRM, ERP 등 다양한 외부 시스템과 연동해야 한다. 이러한 도구들을 안전하게 관리하고, 적절한 권한으로 접근하는 것이 과제이다.

### 6.4 권한 및 정책 관리

아무나 에이전트를 실행할 수 있으면 안 되고, 역할에 따른 권한 설정이 필요하다. 예를 들어 인턴이 사용하는 에이전트가 천만 원짜리 환불을 승인하는 것은 허용되어서는 안 된다.

### 6.5 관측성(Observability)

에이전트가 왜 특정 방식으로 동작했는지 추적하고 분석할 수 있어야 한다. 이것이 불가능하면 에이전트를 개선하거나 문제를 해결하는 것이 매우 어려워진다.

## 7. Amazon Bedrock AgentCore

### 7.1 개요

AgentCore는 위에서 언급한 다섯 가지 엔터프라이즈 요구사항을 모두 해결할 수 있는 **에이전트 운영 플랫폼**이다. 핵심 컴포넌트는 다음의 일곱 가지로 구성된다.

| 컴포넌트 | 역할 |
|---------|------|
| **Runtime** | 세션 격리와 안전한 실행 환경 제공 |
| **Memory** | 단기 및 장기 메모리 제공 |
| **Gateway** | 도구와 데이터 연결 관리 |
| **Identity** | 인증과 권한 관리 |
| **Observability** | 실시간 추적과 분석 |
| **Policy** | 에이전트 행동 바운더리 설정 |
| **Evaluation** | 에이전트 응답 품질 평가 |

중요한 점은 이 모든 기능을 한 번에 도입할 필요 없이, 필요한 서비스만 골라서 사용할 수 있다는 것이다. 또한 어떤 모델이든, 어떤 프레임워크(Strands, LangGraph 등)든 상관없이 사용할 수 있다.

### 7.2 Runtime: 서버리스 에이전트 실행 환경

#### 서버리스 배포

AgentCore Runtime은 서버리스 방식으로 에이전트를 배포하므로 인프라 관리가 따로 필요하지 않다. 스케일링도 AWS가 자동으로 처리한다. 기본 에이전트 코드에 세 줄만 추가하면 가장 기본적인 방식으로 AgentCore Runtime에 배포가 가능하며, 최대 8시간 동안 서버리스 환경에서 실행될 수 있다.

배포 워크플로우는 CLI 기반 툴킷을 통해 진행된다. 프로젝트 생성 → 로컬 서버에서 테스트 → AgentCore Runtime에 배포의 단계를 거친다.

#### 마이크로VM 기반 세션 격리

일반적인 컨테이너 방식에서는 여러 고객의 세션이 같은 커널을 공유하게 된다. 커널 취약점이 발생하면 다른 세션의 데이터가 노출될 위험이 존재한다. LLM 기반 서비스는 코드 실행, 프롬프트 인젝션 등 다양한 공격 벡터가 존재하기 때문에 보안 취약점이 더 커질 수 있다.

AgentCore Runtime은 각 세션마다 독립된 **마이크로VM**을 생성한다. 독립된 커널과 독립된 메모리로 물리적으로 완전히 분리되기 때문에, 한 고객의 세션 정보가 다른 세션과 절대로 공유되지 않는다.

#### 프레임워크 무관 배포

AgentCore Runtime은 특정 모델이나 프레임워크에 구애받지 않는다. Claude 모델과 Strands Agent SDK로 개발된 에이전트뿐만 아니라, 타사 모델이나 LangGraph 등 다른 프레임워크를 사용한 에이전트도 배포가 가능하다.

### 7.3 Memory: 단기 및 장기 메모리

#### 직접 구현의 어려움

메모리 기능을 직접 구현하려면 별도의 데이터베이스를 구축·관리해야 하고, 장기 기억을 위해서는 대화 내역을 요약하여 저장하고, 새 세션 시작 시 다시 불러오는 로직까지 개발해야 한다. 이는 상당히 번거롭고 복잡한 작업이다.

#### AgentCore Memory의 동작 방식

AgentCore Memory를 사용하면 이러한 복잡한 구현 없이 API 호출만으로 메모리 기능을 구현할 수 있다. 동작 과정은 다음과 같다.

1. 대화가 끝나면 **비동기**로 핵심 정보를 추출하고 요약한다.
2. 요약된 정보를 **장기 메모리**에 저장한다.
3. 다음 대화가 시작될 때 장기 메모리에서 핵심 포인트를 불러온다.
4. 불러온 정보를 **컨텍스트에 주입**하여 이전 대화의 맥락을 유지한 채 새로운 대화를 진행한다.

서버리스 방식이므로 인프라 관리가 필요 없고, 암호화와 액세스 제어도 기본 내장되어 있다.

### 7.4 Gateway: 도구 관리 및 접근 제어

#### 시맨틱 검색 기반 도구 선택

에이전트가 사용하는 도구가 많아질수록 컨텍스트가 길어지고, 에이전트의 도구 선택 정확도도 떨어진다. AgentCore Gateway는 **시맨틱 검색**을 통해 이 문제를 해결한다.

예를 들어 "미팅 예약 방법을 알려줘"라는 질문이 들어왔을 때, 300개 이상의 도구 중에서 관련성이 높은 4개만 선별하여 에이전트에 제공한다. 도구들은 자동으로 인덱싱되며, 에이전트가 필요할 때 동적으로 가져오기 때문에 더 빠르고 정확한 응답이 가능하다.

#### 인증 및 권한 관리

Gateway는 인바운드와 아웃바운드 양방향 인증/권한 관리를 제공한다.

- **인바운드**: 사용자가 에이전트를 호출할 때의 로그인 인증, 배치 작업의 자동 실행 인증, 에이전트 간 호출 시의 인증 등을 처리한다.
- **아웃바운드**: 에이전트가 외부 API를 호출하거나, 다른 시스템과 연동할 때의 인증을 관리한다.

### 7.5 Observability: 관측성

프로덕션 환경에서 에이전트가 왜 특정 응답을 했는지 파악하지 못하면, 개선이 불가능하다. AgentCore Observability는 세 가지 핵심 축을 제공한다.

- **로깅(Logging)**: 대화 내역, 에이전트의 추론 과정 등을 기록한다.
- **메트릭(Metrics)**: 호출 수, 지연 시간, 에러율 등 정량적 지표를 수집한다.
- **트레이스(Trace)**: 에이전트가 어떤 도구를 호출했고, 각 도구를 통해 어떤 액션을 취했는지, 어디서 문제가 발생했는지, 왜 특정 도구를 선택했는지 등의 전체 실행 흐름을 추적한다.

Observability는 AgentCore의 모든 기능들과 통합되어 있어, Runtime과 함께 사용하면 런타임에서의 호출 현황, 에러 상태 등을 종합적으로 관찰할 수 있다.

### 7.6 Evaluation: 에이전트 평가

에이전트를 프로덕션에 배포한 후에는 지속적인 성능 측정이 필수적이다. 하지만 최종 목표 달성 여부, 올바른 도구 사용 여부, 답변 정확성, 대화의 적절성 등을 사람이 일일이 로그를 읽으며 평가하기는 현실적으로 어렵다.

AgentCore Evaluation은 **LLM이 에이전트의 대화 기록을 자동으로 평가**해주는 서비스다. 에이전트의 대화 기록(로그)을 LLM이 읽고, 다음과 같은 기준으로 자동 점수를 매긴다.

- 에이전트가 목표를 정말 잘 달성했는가
- 정확한 정보를 제공했는가
- 적절한 도구를 사용했는가
- 정중하고 적절한 어조로 대화했는가

단순히 점수만 부여하는 것이 아니라, 왜 그 점수가 나왔는지 이유를 요약하여 설명해주기 때문에, 에이전트 개선 방향을 구체적으로 파악할 수 있다.

## 8. AgentCore 활용 3단계

AgentCore의 활용은 에이전트의 생명주기에 따라 크게 세 단계로 나뉜다.

### 빌드(Build) 단계

실제 에이전트를 개발하는 단계다. Strands Agent SDK 등의 프레임워크를 사용하면서, AgentCore의 **Memory**, **Identity**, **Policy**, **Gateway** 서비스를 함께 활용하여 에이전트를 구축한다.

### 배포(Deploy) 단계

개발이 완료된 에이전트를 클라우드에 올리는 단계다. AgentCore **Runtime**을 통해 서버리스 환경에 에이전트를 배포한다.

### 운영(Operate) 단계

실제 서비스를 운영하면서 지속적으로 모니터링하고 개선하는 단계다. **Observability**를 통해 실시간 추적과 분석을 수행하고, **Evaluation**을 통해 에이전트의 응답 품질을 자동 평가한다.
